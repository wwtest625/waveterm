### 6. 160 行的巨型 useEffect — 无法维护 ⚠️ 严重 ✅ 已修复

**文件**: `frontend/app/aipanel/aipanel.tsx` L1130-1290

这个 `useEffect` 依赖 `[coalescedMessages, model]`，内部处理了 7 种不同的逻辑。

**🗣️ 大白话解释**:

`useEffect` 是 React 的"副作用"钩子，意思是"当某些东西变化时，执行某些操作"。正常情况下，一个 `useEffect` 做一件事。

但这里一个 `useEffect` 做了 7 件事：提取任务状态、检测命令交互、管理用户提问、派生工具状态、合并 Agent 运行时、检测安全阻止、派发审批事件。

问题：
- **任何消息变化都触发全部 7 件事**：用户只是发了一条"你好"，7 个逻辑全跑一遍，浪费性能
- **改一个逻辑可能影响其他 6 个**：它们共享同一个 effect，改代码时容易互相干扰
- **调试困难**：effect 里某个 `globalStore.set` 导致了 bug，你得从 160 行里找是哪个

**类比**：就像一个开关同时控制灯、空调、电视、窗帘、门锁、音响、热水器。你只想开灯，结果空调也开了、门也锁了。

**🔧 解决方案**:

拆分为 7 个独立的 useEffect：

```typescript
useEffect(() => {
    const taskState = extractTaskState(coalescedMessages);
    globalStore.set(taskStateAtom, taskState);
}, [coalescedMessages]);

useEffect(() => {
    const interaction = extractCommandInteraction(coalescedMessages);
    if (interaction) globalStore.set(commandInteractionAtom, interaction);
}, [coalescedMessages]);

// ... 其余 5 个同理
```

**✅ 修复方案（已实施）**:

通过 `useMessageAnalysis` hook 将原巨型 effect 拆分为 6 个独立的 useEffect：
1. 任务状态派生 effect — 监听 `messageAnalysis.latestTaskState`
2. 命令交互检测 effect — 监听 `messageAnalysis.latestInteractiveToolUse`
3. 用户提问管理 effect — 监听 `messageAnalysis.latestAsk`
4. 工具状态派生 effect — 监听 `messageAnalysis` 中的工具使用数据
5. 流式完成验证 effect — 监听 `status`、`errorMessage`、`commandInteraction`
6. Agent 运行时同步 effect — 监听 `derivedAgentStatusSnapshot`

每个 effect 独立响应其依赖变化，不再存在"一个开关控制 7 个设备"的问题。

### 7. 事件监听器闭包陈旧 ⚠️ 中等 ✅ 已修复

**文件**: `frontend/app/aipanel/aipanel.tsx` L1422-1428

```typescript
useEffect(() => {
    const keyHandler = keydownWrapper(handleKeyDown);
    document.addEventListener("keydown", keyHandler);
    return () => {
        document.removeEventListener("keydown", keyHandler);
    };
}, []);
```

**🗣️ 大白话解释**:

`useEffect` 的依赖数组是 `[]`，意思是"只在组件挂载时执行一次"。但 `handleKeyDown` 函数可能引用了一些会变化的变量。

因为 effect 只执行一次，`handleKeyDown` 捕获的是**创建那一刻**的变量值。之后变量变了，`handleKeyDown` 里的值还是旧的——这就是"陈旧闭包"。

**类比**：就像你让朋友帮你接电话，只告诉了他一次你的名字。后来你改名了，但朋友接电话时还是用旧名字介绍你。

目前因为 `model` 是单例不会变，所以暂时没出问题。但如果以后 `handleKeyDown` 需要访问任何会变化的 state，就会出 bug。

**✅ 修复方案（已实施）**:

使用 `useRef` 模式避免陈旧闭包，确保 `handleKeyDown` 始终引用最新版本：

```typescript
const handleKeyDownRef = useRef(handleKeyDown);
handleKeyDownRef.current = handleKeyDown;

useEffect(() => {
    const keyHandler = (e: KeyboardEvent) => keydownWrapper(handleKeyDownRef.current)(e);
    document.addEventListener("keydown", keyHandler);
    return () => document.removeEventListener("keydown", keyHandler);
}, []);
```

`handleKeyDownRef.current` 每次渲染都会更新为最新的 `handleKeyDown`，而 effect 内部通过 ref 间接引用，避免了对 `handleKeyDown` 的闭包捕获。

### 8. `as any` 类型断言泛滥 ⚠️ 严重 ✅ 已修复

代码中大量使用 `as any` 绕过类型检查。

**🗣️ 大白话解释**:

TypeScript 的类型系统就像安全带——它防止你在代码里犯低级错误。`as any` 就相当于把安全带剪了。

当你写 `const body: any = {...}`，TypeScript 就不再检查 `body` 里面有什么了。你可以写 `body.nonexistentMethod()`，TypeScript 不会报错，但运行时会崩。

代码里有几十处 `as any`，意味着几十个地方没有类型保护。就像一栋楼几十扇门都没锁，小偷（bug）想进哪间进哪间。

**类比**：就像考试时不检查答案直接交卷。也许你写对了，也许写错了，但你放弃了检查的机会。

**✅ 修复方案（已实施）**:

1. **`recordTEvent` 调用（11 处）**：在 Go 端 `telemetrydata.TEventProps` 结构体中添加了 22 个 `WaveAI*` 字段（带 `json:"waveai:xxx,omitempty"` 标签），运行 `go run ./cmd/generatets` 重新生成 TypeScript 类型，使所有 `recordTEvent` 调用的对象字面量均符合 `TEventProps` 类型，无需 `as any`。

2. **`viewModel as any`（1 处）**：使用 `"sendDataToController" in viewModel` 类型守卫替代 `as any`，在确认属性存在后才进行类型窄化：
   ```typescript
   if (viewModel?.viewType === "term" && "sendDataToController" in viewModel && typeof viewModel.sendDataToController === "function") {
       return { blockId, sendDataToController: (viewModel as ViewModel & { sendDataToController: (data: string) => void }).sendDataToController.bind(viewModel) };
   }
   ```

3. **`(part: any)` 参数类型（1 处）**：使用联合类型 + 类型守卫替代 `any`。

**修复结果**：生产代码中仅剩 1 处 `as any`：`(window as any).WaveAIModel`（调试用，可接受）。测试文件中的 `as any` 保留（测试中类型断言合理）。

### 11. 仅有顶层 ErrorBoundary — 子组件崩溃全盘皆输 ⚠️ 中等 ✅ 已修复

**🗣️ 大白话解释**:

ErrorBoundary 是 React 的"安全网"——组件崩溃时显示一个错误提示，而不是整个页面白屏。

目前只有一个顶层的 ErrorBoundary 包住整个 AI 面板。这意味着：如果某一条消息的渲染崩了（比如数据格式不对），**整个 AI 面板都会显示错误**，而不是只让那条消息显示错误。

**类比**：就像一栋楼只有一个总闸。某个房间的灯短路了，整栋楼都停电了。应该每个房间有独立的保险丝，短路只影响那个房间。

**✅ 修复方案（已实施）**:

在 `aipanelmessages.tsx` 中，将 `TaskTurnCard` 外层的 `NullErrorBoundary`（静默吞掉错误）替换为 `ErrorBoundary` + 可见错误回退：

```tsx
<ErrorBoundary
    key={turn.id}
    fallback={
        <div className="mx-3 my-2 rounded-xl border border-red-500/15 bg-red-500/[0.04] px-3 py-2 text-xs text-red-200/70">
            {t.aipanel.messageRenderError}
        </div>
    }
>
    <TaskTurnCard ... />
</ErrorBoundary>
```

单条消息崩溃时，仅该消息位置显示红色错误提示，不影响其他消息的正常渲染。
