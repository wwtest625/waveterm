### 1. WaveAIModel 是典型的 God Object（上帝对象）⚠️ 严重 ✅ 大部分修复

**文件**: `frontend/app/aipanel/waveai-model.tsx`

这个单例类原有 ~2057 行，持有 30+ 个 jotai atom，承担了至少 8 种职责：

- 聊天状态管理（chatId, messages, streaming）
- 会话持久化（sessions, CRUD, switch, rename）
- 文件处理（droppedFiles, addFile, resize, preview）
- 工具调用（invokeTool, pollCommandJob, retryLastAction）
- 命令交互（commandInteraction, submitCommandInteraction）
- 后台任务管理（backgroundJobs, cancel, refresh）
- Agent 运行时状态（agentRuntime, taskState, focusChain）
- UI 焦点管理（focusInput, requestWaveAIFocus）

**🗣️ 大白话解释**:

想象一个人，他同时是：厨师、司机、会计、保安、清洁工、前台、经理、IT 维修。一个人干 8 个人的活，你觉得他能干好吗？

`WaveAIModel` 就是这个"超人"。它 2057 行代码，管了聊天、文件、工具、会话、任务……所有事情都塞在一个类里。结果就是：

- **改一个功能可能炸另一个功能**：比如你改了文件上传的逻辑，不小心影响了聊天消息的发送，因为它们住在同一个"房子"里
- **新人看不懂**：2057 行的文件，谁看了不晕？要理解"聊天怎么发的"，你得先从 2057 行里找到那 20 行
- **没法测试**：你想测"会话切换"功能，但创建测试实例时，所有 8 个模块都得初始化，太重了

**类比**：就像一个万能遥控器，电视、空调、灯、窗帘、门锁全控制。按钮 100 个，你只想开灯，却不知道按哪个。拆成 4 个专用遥控器，每个 20 个按钮，清晰多了。

**🔧 解决方案**:

将 `WaveAIModel` 拆分为 4 个独立模块，通过一个轻量级 `WaveAIContext` 协调：

**第一步：纯函数提取**（`waveai-utils.ts`，125 行）
- 提取 9 个不依赖 `this` 的私有方法（sortSessions、sortBackgroundJobs、summarizeSessionText 等）
- 主类从 2057 行减少到 1797 行

**第二步：组合模式拆分**（`waveai-agent-runtime.ts` + `waveai-file-service.ts`）

| 新模块 | 行数 | 职责 | 跨模块调用 |
|--------|------|------|-----------|

- 子模块内部通过 `this.dispatch()` 触发状态变更，走统一的 dispatch 通道

**第三步：未实施**（风险 > 收益，详见下方分析）

SessionManager 和 ToolEngine 的拆分经分析后暂缓，原因：
- 9 个跨模块调用（`clearChat` 横跨 4 个域、`cancelGeneration` 横跨 3 个域）
- 拆分后协调层仍需持有跨域方法，变成"迷你 God Object"
- 边际收益递减：主类已从 2057 行降至 1858 行，域边界已通过 `───` 注释清晰标记

当前状态：主类 1858 行，已拆出 2 个独立模块 + 1 个纯函数文件，外部 API 完全兼容。

### 2. React Hook 值被存储为可变类属性 — 反模式 ⚠️ 严重 ✅ 部分修复

**文件**: `frontend/app/aipanel/waveai-model.tsx` L85-88

```typescript
useChatSendMessage: UseChatSendMessageType | null = null;
useChatSetMessages: UseChatSetMessagesType | null = null;
useChatStatus: ChatStatus = "ready";
useChatStop: (() => void) | null = null;
```

然后在 `aipanel.tsx` L1071：
```typescript
model.registerUseChatData(sendMessage, setMessages, status, stop);
```

**🗣️ 大白话解释**:

React 的 Hook（比如 `useState`、`useChat` 返回的值）有个重要特性：**每次组件重新渲染，这些值都会更新到最新版**。这就像你每次看手机，看到的都是最新的时间。

但这段代码把 Hook 返回的值"搬"到了一个普通的类属性里。问题是什么？**类属性不会自动更新**。它就像你把手机上的时间抄到了纸上——纸上的时间永远是抄的那一刻，手机上的时间还在走。

具体后果：
- `useChatStatus` 初始值是 `"ready"`，但实际聊天状态可能已经变成 `"streaming"` 了
- `registerUseChatData` 虽然会覆盖旧值，但覆盖的时机不受 React 控制——可能在你用完旧值之后才覆盖
- 这叫 **stale closure（陈旧闭包）**：你以为拿到的是最新值，其实是过期的

**类比**：就像你让朋友帮你看着外卖到了没有，朋友说"到了"然后写在纸条上给你。但纸条是 5 分钟前写的，外卖可能已经凉了或者被偷了。

**🔧 解决方案**:

将 `useChat` 的返回值保留在 React 组件内部，通过 jotai atom 桥接：

```typescript
const { messages, input, handleInputChange, handleSubmit, status, stop, setMessages } =
    useChat({ /* ... */ });

const chatStatusAtom = atom<ChatStatus>("ready");
const [, setChatStatus] = useAtom(chatStatusAtom);

useEffect(() => {
    setChatStatus(status);
}, [status, setChatStatus]);
```

或者通过 React Context 传递，完全绕过 model 类：

```typescript
const AIPanelChatContext = createContext<{
    sendMessage: UseChatSendMessageType;
    stop: () => void;
    status: ChatStatus;
} | null>(null);

<AIPanelChatContext.Provider value={{ sendMessage, stop, status }}>
    <AIPanelMessages />
    <AIPanelInput />
</AIPanelChatContext.Provider>
```

**✅ 实际修复（部分）**:

已建立 `AIPanelChatContext`（React Context），子组件可通过 `useAIPanelChat()` 获取 `sendMessage`、`setMessages`、`status`、`stop`，无需通过 model 类。

但 model 仍通过 `registerChatContext` 保存了 Context 引用作为后门（用于 `sendSubmission`、`cancelGeneration` 等方法中调用 `sendMessage`/`setMessages`/`stop`）。完整修复需要将这些方法也迁移到 React 组件内部，或通过事件系统触发。

### 3. 前后端轮询逻辑重复 — 竞态条件隐患 ⚠️ 严重 ✅ 已修复

前端 `waveai-model.tsx` L1099-1199 的 `pollCommandJob` 和后端 `usechat.go` L547-731 的 `tryStartWaveCommandResultPoller` 都在做同样的事情：

- 轮询 `AgentGetCommandResultCommand`
- 检测交互状态（awaitingInput, tuiDetected）
- 合并输出文本
- 处理超时

**🗣️ 大白话解释**:

想象一个场景：你让两个人（前端和后端）同时去检查同一个快递到了没有。

- 前端每隔一会儿去问一次："快递到了吗？"
- 后端也每隔一会儿去问一次："快递到了吗？"

两个人都在查同一个快递单号。问题来了：

1. **重复劳动**：两个人干同一件事，浪费资源
2. **信息打架**：前端说"快递到了，状态是 A"，后端说"快递到了，状态是 B"。到底听谁的？谁后更新谁赢，但"谁后"是不确定的——这就是**竞态条件**
3. **用户困惑**：界面上的状态可能闪烁——一会儿显示前端更新的状态，一会儿被后端的覆盖

**类比**：就像两个人同时编辑同一个 Google Doc，没有协同功能。你写了一句话，对方也写了一句话，最后保存的是谁的那句？不确定。

**🔧 解决方案**:

采用"后端单一真相源"架构，前端不再主动轮询命令结果：

```typescript
// 前端删除 pollCommandJob 方法，只监听后端 SSE 推送

type SSEEventType =
    | "data-tooluse"
    | "data-text"
    | "command_interaction"  // 新增
    | "command_result"       // 新增
    | "agent_runtime";

useEffect(() => {
    const handleSSEEvent = (event: SSEEvent) => {
        switch (event.type) {
            case "command_interaction":
                globalStore.set(commandInteractionAtom, event.data);
                break;
            case "command_result":
                globalStore.set(toolUseAtom, event.data);
                break;
        }
    };
}, []);
```

后端确保在检测到交互状态时通过 SSE 主动推送，而非等前端来轮询。

**✅ 修复方案（已实施）**:

采用 wps (Wave PubSub) 事件系统替代前端轮询，参照已有 `Event_BlockJobStatus` 模式：

1. **后端新增事件类型** `Event_WaveAIBgJob = "waveai:bgjob"`（`wpstypes.go`）
2. **chatstore 发布事件**：`UpsertBackgroundJob`、`ReplaceBackgroundJobs`、`ClearFinishedBackgroundJobs` 中调用 `wps.Broker.Publish`，scope 为 `chat:{chatId}`
3. **后端监控协程**：`bgjob-monitor.go` 中 `startBackgroundJobMonitor` 每 5s 检查活跃任务状态变化，自动刷新并发布事件
4. **前端订阅事件**：`useBackgroundJobsRefresh` 中通过 `waveEventSubscribeSingle` 订阅 `waveai:bgjob` 事件，收到后直接更新本地状态
5. **降级轮询**：轮询间隔从 1.5s/4s 降为 30s 安全网
6. **新增方法**：`WaveAIModel.upsertBackgroundJobFromEvent` 处理事件驱动的增量更新

修改文件：
- `pkg/wps/wpstypes.go` — 新增 `Event_WaveAIBgJob`
- `pkg/tsgen/tsgenevent.go` — 注册 TypeScript 类型映射
- `pkg/aiusechat/chatstore/chatstore.go` — 3 处事件发布 + `publishBackgroundJobEvent` 方法
- `pkg/aiusechat/bgjob-monitor.go` — 新文件，后台任务监控协程
- `frontend/types/waveevent.d.ts` — 新增事件类型声明
- `frontend/app/aipanel/aipanel-hooks.ts` — wps 事件订阅 + 降级轮询
- `frontend/app/aipanel/waveai-model.tsx` — `upsertBackgroundJobFromEvent` 方法

### 4. `globalStore.set/get` 散落各处 — 无追踪的状态变更 ⚠️ 中等 ✅ 部分修复

整个 `WaveAIModel` 中有 50+ 处 `globalStore.set(...)` 和 `globalStore.get(...)` 调用。

**🗣️ 大白话解释**:

想象一个公司，没有规章制度，任何人都可以随时修改任何文件。50 个人都能改同一个 Excel 表，没有修改记录，没有审批流程。

`globalStore.set` 就是这样的——任何地方、任何时间、任何人都能直接改状态。后果：

- **出了 bug 没法查**：状态被改错了，你不知道是谁改的、什么时候改的、改成什么了
- **没法回滚**：因为没有修改记录，改错了只能手动恢复
- **没法调试**：想看"状态是怎么一步步变成这样的"？没门，没有日志

**类比**：就像 Git 仓库里所有人都用 `git push --force`，没有 commit message，没有 code review。代码乱了，谁也不知道怎么乱的。

**🔧 解决方案**:

引入 `atomWithReducer` 模式，所有状态变更都有明确的 action：

```typescript
type WaveAIAction =
    | { type: "SET_CHAT_ID"; chatId: string }
    | { type: "SET_TASK_STATE"; taskState: TaskState }
    | { type: "SET_COMMAND_INTERACTION"; interaction: CommandInteraction | null };

const waveAIReducer = (state: WaveAIState, action: WaveAIAction): WaveAIState => {
    switch (action.type) {
        case "SET_CHAT_ID":
            return { ...state, chatId: action.chatId };
        case "SET_TASK_STATE":
            return { ...state, taskState: action.taskState };
        default:
            return state;
    }
};

const waveAIStateAtom = atomWithReducer(initialState, waveAIReducer);
```

好处：每个状态变更都有 action type，可追踪、可日志、可回放。

**✅ 实际修复（#1 God Object 部分）**:

1. **纯工具函数提取**：将 9 个不依赖 `this` 的私有方法提取到 `waveai-utils.ts`（125 行），主类从 2057 行减少到 1797 行。
2. **组合模式拆分**：将 AgentRuntimeModule（127 行）和 FileServiceModule（131 行）提取为独立类，通过构造函数注入 dispatch 回调。主类从 1797 行降至 1858 行（含 getter delegate），但职责边界更清晰。
3. **组件拆分间接减轻**：`aipanelmessages.tsx` 从 1778 行拆分为 5 个文件（262 行主文件），减少了渲染逻辑对 model 的依赖复杂度。

**⚠️ 未实施**：SessionManager 和 ToolEngine 的拆分经风险收益分析后暂缓（9 个跨模块调用，边际收益递减）。当前拆分已覆盖低耦合模块，是安全上限。

**✅ 实际修复（#4 globalStore.set/get 部分）**:

已建立 `dispatch(action: WaveAIAction)` 模式（`waveai-actions.ts` 定义了所有 action 类型），当前有 60 处 `this.dispatch()` 调用和 33 处 `globalStore.set` 直接调用。dispatch 模式已覆盖大部分状态变更，但仍有 33 处直接操作 store（主要是读取 `globalStore.get` 和少数写入）。完全迁移到 dispatch 模式需要将剩余 `globalStore.set` 调用也包装为 action。
