### 13. 全局可变状态 + 手动互斥锁 — 易出错 ⚠️ 严重 ✅ 已修复

**文件**: `pkg/aiusechat/usechat.go`

5 组全局可变状态，各自有独立的锁，没有统一的并发模型。

**🗣️ 大白话解释**:

后端代码里有 5 个"全局变量"，每个都配了一把"锁"（Mutex）。多个 goroutine（可以理解为线程）同时访问这些变量时，必须先锁上，用完再解锁。

问题：

1. **锁的顺序不统一**：A 函数先锁变量 1 再锁变量 2，B 函数先锁变量 2 再锁变量 1。如果 A 和 B 同时运行，A 等变量 2 的锁，B 等变量 1 的锁——谁也等不到谁，这就是**死锁**，程序卡死
2. **没法测试**：全局变量意味着所有测试共享同一份数据。测试 A 改了全局变量，测试 B 就受影响。你没法"隔离"每个测试
3. **没法清理**：程序运行久了，全局变量里可能积累了大量过时数据，没有统一的清理机制

**类比**：就像 5 个仓库，每个仓库一把钥匙，5 个工人各自拿钥匙。没人规定"必须先开 1 号仓库再开 2 号仓库"，结果两个人互相等对方手里的钥匙，谁也进不去。

**🔧 解决方案**:

封装为结构体，统一生命周期管理：

```go
type ChatManager struct {
    rateLimitInfo *uctypes.RateLimitInfo
    rateLimitLock sync.Mutex
    activeChats   *ds.SyncMap[bool]
    tuiAutoCancelMu      sync.Mutex
    tuiAutoCancelledJobs map[string]bool
    commandPollerMu sync.Mutex
    commandPollers  map[string]struct{}
    commandJobMu    sync.Mutex
    commandJobs     map[string]waveCommandJobEntry
}

func NewChatManager() *ChatManager {
    return &ChatManager{
        rateLimitInfo:        &uctypes.RateLimitInfo{Unknown: true},
        activeChats:          ds.MakeSyncMap[bool](),
        tuiAutoCancelledJobs: make(map[string]bool),
        commandPollers:       make(map[string]struct{}),
        commandJobs:          make(map[string]waveCommandJobEntry),
    }
}
```

**✅ 实际修复**:

- 定义 `ChatManager` 结构体，将 5 组全局可变状态（`globalRateLimitInfo`/`rateLimitLock`、`activeChats`、`tuiAutoCancelMu`/`tuiAutoCancelledJobs`、`waveCommandResultPollers`、`waveCommandJobContext`）封装为结构体字段
- 创建 `defaultChatManager` 全局实例，通过 `NewChatManager()` 初始化
- 将状态操作实现为 `ChatManager` 方法：`updateRateLimit()`、`getRateLimit()`、`cleanupCommandJobsLocked()`、`rememberCommandJob()`、`lookupCommandJob()`
- 旧自由函数（`updateRateLimit`、`GetGlobalRateLimit`、`rememberWaveCommandJob`、`lookupWaveCommandJob`、`cleanupWaveCommandJobsLocked`）改为薄包装，委托到 `defaultChatManager`
- `autoCancelTUICommand`、`tryStartWaveCommandResultPoller`、`RunAIChat` 直接访问 `defaultChatManager` 字段
- 测试文件 `usechat_wave_command_context_test.go` 同步更新为 `defaultChatManager` 引用

### 14. HTTP Handler 缺少认证 ⚠️ 中等   PASS 不做

**🗣️ 大白话解释**:

后端的 HTTP 接口（比如发送消息的接口）没有任何身份验证。任何人只要知道端口号，就能发请求。

目前 waveterm 是本地应用，只有你自己能用，所以暂时不是大问题。但如果未来支持远程访问或多人使用，这就是一个安全漏洞——别人可以冒充你发消息。

**类比**：就像一栋房子没有门锁。现在只有你一个人住，没问题。但如果以后有室友或客人，谁都能进你的房间。

**🔧 解决方案**:

添加基于 session token 的认证中间件。对于本地应用，可用进程启动时生成的随机 token。

### 15. 验证错误返回 500 而非 400 ⚠️ 中等 ✅ 已修复

**文件**: `pkg/aiusechat/usechat.go`

```go
if err := req.Msg.Validate(); err != nil {
    http.Error(w, fmt.Sprintf("Message validation failed: %v", err), http.StatusInternalServerError)
    return
}
```

**🗣️ 大白话解释**:

HTTP 状态码有个约定：

- **400 Bad Request** = "你（客户端）发的东西有问题"
- **500 Internal Server Error** = "我（服务器）内部出错了"

消息验证失败，是客户端的问题（比如消息格式不对），应该返回 400。但代码返回了 500，等于告诉客户端"服务器出错了"。

这会导致：

- 客户端以为是服务器 bug，不断重试（其实重试也没用，因为请求本身就有问题）
- 监控系统误报：500 错误被当作服务器故障报警，运维半夜被叫起来排查，结果发现只是客户端传了个空消息

**类比**：就像快递员说"包裹送不到"，原因是地址写错了（400），但他告诉你"我们仓库着火了"（500）。你打电话投诉仓库，但仓库没问题啊。

**✅ 实际修复**:

```go
if err := req.Msg.Validate(); err != nil {
    http.Error(w, fmt.Sprintf("Message validation failed: %v", err), http.StatusBadRequest)
    return
}
```

将 `http.StatusInternalServerError` 改为 `http.StatusBadRequest`。

### 16. Panic recovery 吞掉严重 bug ⚠️ 中等 ✅ 已修复

**🗣️ 大白话解释**:

Go 语言里 `panic` 类似其他语言的异常，通常意味着严重的编程错误（比如访问了 nil 指针、数组越界）。

代码里有个 `recover()` 把 panic 捕获了，然后只返回一个错误消息给用户。这等于把 bug **藏起来了**——用户看到"工具执行出错"，但开发者完全不知道发生了 panic。

**类比**：就像汽车的安全气囊弹出来了，但仪表盘上只显示"请检查车辆"，不告诉你气囊弹了。你继续开，下次可能更严重。

**✅ 实际修复**:

Panic recovery 现在记录完整堆栈并提供明确的错误提示：

```go
defer func() {
    if r := recover(); r != nil {
        buf := make([]byte, 4096)
        n := runtime.Stack(buf, false)
        stackTrace := string(buf[:n])
        log.Printf("PANIC in tool execution: %v\nStack:\n%s", r, stackTrace)
        result.ErrorText = fmt.Sprintf("Internal error (panic: %v). This is a bug, please report it.", r)
        result.Text = ""
    }
}()
```

- 添加 `runtime.Stack()` 调用捕获完整堆栈
- 使用 `log.Printf` 输出 panic 信息和堆栈到日志
- 错误消息改为 "Internal error (panic: %v). This is a bug, please report it."，明确告知用户这是 bug
- 添加 `"runtime"` import

### 17. processToolCallBatch 的伪并行 ⚠️ 严重 ✅ 已修复

**🗣️ 大白话解释**:

代码用 `sync.WaitGroup` 启动了多个 goroutine 来并行执行工具调用，看起来是并行的。但问题是：

1. **所有 goroutine 必须全部完成后，才能开始处理结果**（`wg.Wait()` 在 `range resultCh` 之前）。就像 5 个人同时跑 100 米，但必须等最后一个人跑完才能宣布成绩——跑得快的人白快了
2. **没有取消机制**：如果 SSE 连接断了（用户关了页面），所有工具调用还在继续执行，浪费资源
3. **结果不流式**：用户要等所有工具都执行完才能看到任何结果

**类比**：就像去餐厅点 5 道菜，5 个厨师同时做。但服务员必须等 5 道菜全做好才上桌。第一道菜 1 分钟就好了，但最后一道菜要 30 分钟——你饿着肚子等了 30 分钟，5 道菜一起上，前 4 道都凉了。

**✅ 实际修复**:

1. **流式处理**：将 `wg.Wait()` → `close(resultCh)` → `range` 改为后台 goroutine 关闭 channel，主 goroutine 直接 `range resultCh` 流式处理结果。先完成的工具调用先被 `finalizeToolCallProcessing` 处理，不再等待所有 goroutine 完成
2. **context 传播**：`processToolCallBatch` 和 `processAllToolCalls` 添加 `ctx context.Context` 参数。goroutine 启动前检查 `ctx.Err()`，若已取消则直接返回 cancelled 结果；`processAllToolCalls` 的执行计划循环中添加 `ctx.Err()` 检查
3. **调用链更新**：`RunAIChat` → `processAllToolCalls(ctx, ...)` → `processToolCallBatch(ctx, ...)`，context 从顶层传递到底层

```go
func processToolCallBatch(
    ctx context.Context,
    backend UseChatBackend,
    toolCalls []uctypes.WaveToolCall,
    chatOpts uctypes.WaveChatOpts,
    sseHandler *sse.SSEHandlerCh,
    metrics *uctypes.AIMetrics,
) []uctypes.AIToolResult {
    // ...
    go func() {
        defer wg.Done()
        if ctx.Err() != nil {
            resultCh <- batchResult{
                index:  index,
                result: uctypes.AIToolResult{ToolName: call.Name, ToolUseID: call.ID, ErrorText: "cancelled"},
                toolCall: call,
                toolDef:  def,
            }
            return
        }
        // ... execute tool call
    }(idx, toolCall, toolDef)

    go func() {
        wg.Wait()
        close(resultCh)
    }()

    for item := range resultCh {
        results[item.index] = item.result
        finalizeToolCallProcessing(backend, chatOpts, sseHandler, item.toolCall, item.toolDef, item.result, metrics)
    }
    return results
}
```

### 18. waveCommandJobContext 内存泄漏风险 ⚠️ 轻微 ✅ 已修复

**🗣️ 大白话解释**:

`waveCommandJobContext` 是一个全局 map，存储命令执行任务的上下文。清理逻辑只在有新命令创建时才执行。如果长时间没有新命令，旧条目会一直留在内存中。

**类比**：就像一个停车场，只有新车进来时才清理旧车位。如果很久没有新车进来，已经开走的车位永远不会被标记为空。

**✅ 实际修复**:

添加 `init()` 函数启动定期清理 goroutine，每 5 分钟执行一次 `cleanupCommandJobsLocked`：

```go
const waveCommandJobCleanupInterval = 5 * time.Minute

func init() {
    go func() {
        ticker := time.NewTicker(waveCommandJobCleanupInterval)
        defer ticker.Stop()
        for range ticker.C {
            defaultChatManager.commandJobMu.Lock()
            defaultChatManager.cleanupCommandJobsLocked(time.Now())
            defaultChatManager.commandJobMu.Unlock()
        }
    }()
}
```

清理逻辑复用已有的 `cleanupCommandJobsLocked` 方法（清理超过 `waveCommandJobRetention=60min` 的旧条目，并限制总数不超过 `waveCommandJobMaxCount=2048`）。
