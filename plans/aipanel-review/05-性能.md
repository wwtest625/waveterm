### 26. `JSON.stringify` 用于深度比较 — 不可靠且低效 ⚠️ 中等 ✅

**🗣️ 大白话解释**:

代码用 `JSON.stringify(a) !== JSON.stringify(b)` 来判断两个对象是否相同。

问题 1——**不可靠**：`JSON.stringify` 的输出依赖 key 的顺序。`{a:1, b:2}` 和 `{b:2, a:1}` 内容一样，但 stringify 的结果不同，会被误判为"不同"。

问题 2——**低效**：每次比较都要把整个对象转成字符串，大对象的字符串化很慢。

**类比**：就像通过"把两篇文章逐字抄一遍然后对比"来判断文章是否相同。既慢又不准——同一篇文章换个段落顺序，抄出来就不一样了。

**🔧 实际修复**:

用内联浅比较数组替换 `JSON.stringify`，两处：

1. `waveai-model.tsx` — `activeJobIds` 比较：
```typescript
// Before
JSON.stringify(current.activeJobIds ?? []) !== JSON.stringify(next.activeJobIds ?? [])
// After
const curIds = current.activeJobIds ?? [];
const nxtIds = next.activeJobIds ?? [];
curIds.length !== nxtIds.length || curIds.some((id, i) => id !== nxtIds[i])
```

2. `aipanel.tsx` — `inputOptions` 比较：
```typescript
// Before
JSON.stringify(currentInteraction?.inputOptions ?? []) !== JSON.stringify(nextInteraction.inputOptions ?? [])
// After
const curOpts = currentInteraction?.inputOptions ?? [];
const nxtOpts = nextInteraction.inputOptions ?? [];
curOpts.length !== nxtOpts.length || curOpts.some((opt, i) => opt !== nxtOpts[i])
```

### 27. `coalescedMessages` 每次渲染都创建新对象 ⚠️ 严重 ✅

**文件**: `frontend/app/aipanel/aipanel.tsx` + `frontend/app/aipanel/aitypes.ts`

**🗣️ 大白话解释**:

AI 流式输出时，消息每 34ms 更新一次。每次更新，这段代码都会对**所有消息**执行 `map` + 浅拷贝 + `coalesceMessageParts`。

假设有 20 条消息，只有最后 1 条在变化，但 20 条全部被重新处理。这就像每次只改了一个字，却把整本书重新排版印刷一遍。

**类比**：就像一个 100 人的班级，1 个学生改了名字，老师把 100 个人的花名册全部重新抄一遍。其实只需要改那 1 个人的记录。

**🔧 实际修复**:

两层优化，避免不必要的对象创建：

1. `coalesceMessageParts`（aitypes.ts）：当没有重复 tool detail parts 时返回原数组（引用相等）：
```typescript
export function coalesceMessageParts(parts: WaveUIMessagePart[]): WaveUIMessagePart[] {
    let hasCoalesced = false;
    const result: WaveUIMessagePart[] = [];
    const toolDetailIndexByKey = new Map<string, number>();
    for (const part of parts) {
        if (isToolDetailPart(part)) {
            const key = `${part.type}:${part.data.toolcallid}`;
            const existingIndex = toolDetailIndexByKey.get(key);
            if (existingIndex != null) {
                result[existingIndex] = part;
                hasCoalesced = true;
            } else {
                toolDetailIndexByKey.set(key, result.length);
                result.push(part);
            }
        } else {
            result.push(part);
        }
    }
    return hasCoalesced ? result : parts;  // 无合并时返回原数组
}
```

2. `coalescedMessages` useMemo（aipanel.tsx）：只在 coalescing 实际改变 parts 时才创建新 message 对象：
```typescript
const coalescedMessages = useMemo(() => {
    let changed = false;
    const result: typeof messages = [];
    for (const msg of messages) {
        const coalescedParts = coalesceMessageParts(msg.parts);
        if (coalescedParts === msg.parts) {
            result.push(msg);           // 复用原 message 对象
        } else {
            changed = true;
            result.push({ ...msg, parts: coalescedParts });
        }
    }
    return changed ? result : messages;  // 全部未变则返回原数组
}, [messages]);
```

### 28. `refreshTerminalTargetInfo` 每秒轮询 ⚠️ 中等 ✅

**🗣️ 大白话解释**:

AI 面板打开时，代码每秒读取一次终端目标信息（当前焦点的 block），然后更新状态。

问题：终端目标大部分时间是不变的。你盯着同一个终端看，它不会变。但代码每秒都去读、都去更新状态，即使什么都没变。

每秒一次看似不多，但：

- 每次更新都可能触发 React 重新渲染
- 如果有多个组件订阅了这个状态，每秒都会重新渲染
- 电池供电的设备上，不必要的唤醒会消耗电量

**类比**：就像你每秒钟看一次手表，即使你根本不赶时间。看了 3600 次，只有 2 次是有意义的。

**🔧 实际修复**:

改为**事件驱动 + 长间隔回退**的混合方案（`aipanelinput.tsx`）：

1. 订阅 `FocusManager.blockFocusAtom`，焦点 block 变化时立即刷新
2. 保留 10 秒间隔轮询作为 CWD 等元数据变化的后备（从 1 秒延长到 10 秒）
3. 轮询频率降低 90%，焦点变化时零延迟响应

```typescript
const focusedBlockId = useAtomValue(FocusManager.getInstance().blockFocusAtom);

// 长间隔回退（10秒而非1秒）
useEffect(() => {
    if (!isPanelOpen) return;
    model.refreshTerminalTargetInfo();
    const intervalId = window.setInterval(() => {
        model.refreshTerminalTargetInfo();
    }, 10000);
    return () => window.clearInterval(intervalId);
}, [isPanelOpen, model]);

// 事件驱动：焦点 block 变化时立即刷新
useEffect(() => {
    if (!isPanelOpen || focusedBlockId == null) return;
    model.refreshTerminalTargetInfo();
}, [isPanelOpen, focusedBlockId, model]);
```

### 29. 上下文 token 估算极度粗糙 ⚠️ 中等 ✅

**修复方案**: 区分 CJK/拉丁字符的混合估算

- CJK 字符（中日韩）：约 0.7 token/字符
- 拉丁/其他字符：约 1 token/4 字符
- 覆盖 Unicode 范围：CJK 统一汉字、平假名/片假名、韩文、扩展汉字、兼容汉字
- 比原来的 `length / 3` 对中文和代码都更准确

### 30. Background Jobs 轮询间隔不合理 ⚠️ 轻微 ✅

**🗣️ 大白话解释**:

后台任务的轮询间隔取决于当前是否有活跃任务：有活跃任务时 1.5 秒轮询一次，没有时 4 秒一次。

但 `useEffect` 的依赖包含 `backgroundJobs`，每次刷新都返回新的数组 → 触发 effect 重执行 → 清除旧 timer → 创建新 timer。

结果：不是"每 1.5 秒刷新一次"，而是"刷新后等 1.5 秒再刷新"。如果某次刷新花了 0.5 秒，实际间隔是 2 秒。

**类比**：就像你设了闹钟每小时响一次，但每次响完你重新设闹钟。如果处理闹钟花了 5 分钟，实际间隔变成了 65 分钟。

**🔧 实际修复**:

将 `backgroundJobs` 数组依赖替换为 `hasActiveJobs` 布尔值依赖（`aipanel-hooks.ts`）：

```typescript
export function useBackgroundJobsRefresh(...) {
    const timerRef = useRef<number | null>(null);
    const hasActiveJobs = backgroundJobs.length > 0
        && backgroundJobs.some((job) => !isTerminalBackgroundJobStatus(job.status));

    // 初始刷新（不变）
    useEffect(() => {
        if (!chatIdValue || !isPanelVisible) return;
        void model.refreshBackgroundJobs(chatIdValue);
    }, [chatIdValue, isPanelVisible, model]);

    // 轮询：只在 hasActiveJobs 布尔值变化时重建 timer
    useEffect(() => {
        if (timerRef.current != null) {
            window.clearInterval(timerRef.current);
            timerRef.current = null;
        }
        if (!chatIdValue || !isPanelVisible || backgroundJobs.length === 0) return;
        const interval = hasActiveJobs ? 1500 : 4000;
        timerRef.current = window.setInterval(() => {
            void model.refreshBackgroundJobs(chatIdValue);
        }, interval);
        return () => {
            if (timerRef.current != null) {
                window.clearInterval(timerRef.current);
                timerRef.current = null;
            }
        };
    }, [chatIdValue, isPanelVisible, hasActiveJobs, model]);  // hasActiveJobs 而非 backgroundJobs
}
```

关键变化：依赖从 `backgroundJobs`（每次刷新都是新引用）改为 `hasActiveJobs`（布尔值，只在活跃状态切换时变化），timer 不再因每次刷新结果而重建。
