# 网络侧边栏亮点版设计方案

## 背景

现有方案已经能把网卡信息看清楚，也能指导实现，但整体还是偏工具面板。它解决了可用性，却没有把网络这件事从参数管理提升成状态感知。打开后看到的仍是一组网卡条目，而不是这台机器当前网络状况的结论。

这版设计的目标不是多加几个按钮，而是换一个视角。网络侧边栏不该先展示接口列表，而该先告诉用户现在网络怎么样，主连接是谁，异常在哪里，哪些接口值得关注。列表是证据层，不是首页结论层。

## 新的产品定义

把网络侧边栏从网卡列表工具改成网络状态中心。

用户打开面板后的第一感受，不应是这里有 6 张网卡，而应是当前网络正常，主出口是 `en0`，本机地址是多少，DNS 是否健康，最近有没有异常波动。只有在需要深挖时，才下钻到具体网卡。

## 核心设计思路

整体改成三层。

最上层是全局结论区，用一句状态结论加几个最关键指标，直接回答网络是否正常。

中间层是主连接卡，突出当前真正承载流量的接口，把它从一堆网卡里拎出来。

下层才是接口列表和虚拟网络分组，让信息完整但不喧宾夺主。

这样做的结果，是用户先得到判断，再看到结构，最后才看细节。阅读路径会比传统折叠列表顺很多。

## 布局方案

侧边栏顶部做成一块紧凑的网络总览卡。这里不讲参数全集，只给结论和关键状态。建议显示网络总状态、主连接名称、主 IPv4、默认网关、DNS 状态、近一分钟流量趋势。如果存在明显异常，顶部卡片直接变成异常态，并显示一句短诊断。

总览卡下面放主连接卡。只展示一张当前主接口卡，把最常用的那张网卡单独拉出来。卡片上直接显示接口名、连接类型、有无线则显示 SSID 和信号强度、有线则显示速率，外加一个很轻的上下行流量动态。这样用户打开就知道当前到底靠谁在联网。

主连接卡下面放接口分组列表。这里不再简单按顺序平铺所有网卡，而是分成活动接口、其他物理接口、虚拟接口三组。虚拟接口默认折叠，避免 `docker0`、`br-*`、`veth*` 把真实网卡淹没。分组标题直接显示数量。

选中某张接口后，不建议在原地展开成一大坨。更合适的是右侧主区显示网卡剖面页，或者在当前面板下半区切换成详情页。这样窄栏里不会被表单和长字段撑爆，信息密度也更可控。

## 第一眼要看到什么

顶部结论卡建议固定输出一句机器可读也人可读的话。

正常时可以是 当前网络正常，主连接为 `enp3s0`。

异常时可以是 已获取 IP，但默认网关不可达。

或者 DNS 响应异常，域名解析可能失败。

这类结论不是文案装饰，它应该由底层规则判断生成。只要这层做对，整个侧边栏的产品气质就会立刻*不一样。用户不再需要自己读一遍网*卡数据后脑补结论。

## 让人眼前一亮的关键能力

最值得做的是轻诊断，不是重配置。

侧边栏应该自动识别几类高频网络异常。比如接口已连接但没有 IP，拿到了 IP 但没有默认路由，默认路由存在但 DNS 无响应，同时存在多个默认路由导致出口混乱，当前主接口长期无流量，虚拟网卡数量过多掩盖真实接口。检测到后，不需要弹大告警，只要在顶部给出一句清楚结论，并在对应接口旁打异常标记。

第二个亮点是主连接识别。不要让所有网卡平权。系统里真正值得用户关心的，通常就那一两张。可以根据默认路由、当前流量、链路状态综合判断当前主连接，并固定展示在总览区。这个动作会让界面显得聪明，而不是被动罗列字段。

第三个亮点是接口角色感。每张网卡不只显示它是什么，还显示它在系统里扮演什么角色。比如主出口、局域网接口、容器桥接、VPN 隧道、闲置接口。角色一旦出现，用户理解成本会大幅下降。

第四个亮点是轻量动态感。网络不是静态配置，它天然适合少量活的视觉反馈。总览卡可以有极轻的流量波形，主连接卡可以有微弱流动背景，异常状态可以让状态条出现断裂感。只要足够克制，就会比纯表格式信息更有生命力。

## 交互改法

现有展开式列表太像设置页。亮点版建议改成选中式交互。

左侧只负责选择对象和显示总体状态，不承担所有细节。用户点中某张接口时，界面进入该接口视角，看到它的地址、路由、DNS、驱动、MTU、速率、最近流量，以及它参与了哪些系统网络关系。

复制动作仍然保持一键完成，但危险操作要继续收进更多菜单。启用、禁用、重启这类动作不要成为第一视觉层的主元素。真正高频的是看和判断，不是改。

虚拟网卡建议默认弱化显示。不是隐藏，而是收拢成一个虚拟网络组。用户有需要时再展开。否则 Docker、Podman、VPN 之类环境一多，整个面板就会非常吵。

## 视觉方向

这版不建议继续照着 Docker Desktop 做。可以借它的整洁，但不要借它的平。网络面板更适合有一点监控感和脉冲感。

色彩上还是跟 WaveTerm 主题走，但可以加入一种偏青蓝的网络强调色，让 IP、链路状态、趋势图有统一识别。正常态尽量平静。异常态不靠整块红色吓人，而是用局部高亮和一句短结论制造注意力。

排版上建议减少同权重文本，强化大小对比。顶部结论最大，主 IP 次之，接口名和状态标签再下一层，技术参数全部退后。这样扫描路径才清楚。

## 推荐的信息层级

用户进入侧边栏后，应该自然经历这条路径。

先看到 当前网络是否正常。

再看到 现在主要通过哪张网卡联网。

再看到 还有哪些接口在线，哪些只是背景噪音。

最后才进入 某张接口的详细配置和操作。

只要把这个顺序守住，整个产品就会从字段浏览器变成状态决策工具。

## 可以直接落地的版本

如果想控制实现成本，不必一次做满。

第一步就改布局，**把顶部总览卡和主连接卡加上，下面列表按活动接口和虚拟接口分组。这一步**不需要引入复杂后端，收益已经很明显。

第二步补轻诊断规则，只先做最有价值的几类。像默认路由缺失、DNS 异常、接口已连接但无地址，这几类就足够让产品开始发亮。

第三步再补趋势和动态视觉，把总览卡真正做成有状态感的面板。

## 一个更像成品的界面印象

打开侧边栏，最上面先看到一张窄但醒目的总览卡。卡片告诉你当前网络正常，主连接是 `wlan0`，SSID 是 `Office-5G`，IP 是 `192.168.31.24`，DNS 正常，流量轻微波动。

下面是一张主连接卡，像驾驶舱里的主仪表。信号、速率、上下行趋势都在这里。

再往下才是其他接口。有线口显示未连接，Docker 网桥收在虚拟网络组里，VPN 隧道如果在线则单独标出。

一眼看过去，不用展开列表，不用读完整参数，机器当前的网络状态已经清楚了。

## 第一步落地稿

第一步只做结构升级，不做复杂诊断，不做趋势图，不改危险操作链路。目标只有一个，打开侧边栏时，先看到网络结论和主连接，再看到整理过的接口列表。

页面从上到下改成三块。第一块是网络总览卡，第二块是主连接卡，第三块是接口分组列表。这样第一屏先回答网络是否正常，第二屏再进入接口明细。

总览卡保持紧凑，只放最值得先看的五个信息，当前状态、主连接、IP 地址、默认网关、DNS 状态。当前状态首版先收敛成三种，正常、受限、离线。正常表示存在可用主连接，且有 IP、默认网关和 DNS。受限表示接口在线，但配置不完整。离线表示没有可用物理接口，或者没有拿到地址。

主连接卡只展示一个接口。优先取默认路由命中的接口，没有默认路由时，再退回第一个 `up` 且带 IPv4 的物理接口。卡片里显示接口名、类型、主 IP、链路状态，再补一条很短的辅助说明，比如 有默认路由 或 未检测到网关。这里不放禁用、重启这类动作，只保留进入详情和复制地址。

接口列表不再平铺，而是分成活动接口、其他接口、虚拟网络三组。活动接口指当前在线，且有地址或承担主连接角色的接口。其他接口主要是物理接口里的待机项。虚拟网络默认折叠，只显示标题和数量，展开后再看明细。这样 Docker、桥接、隧道接口不会把真实网络信息冲散。

列表项默认态要收敛。只显示接口名、状态、主 IP 和角色标签。角色标签首版只做三种，主连接、活动、虚拟。MAC、MTU、速率这些信息全部退到详情里。这样顶部两张卡才能成为真正的视觉中心。

## 第一步实现边界

这一步不需要新后端大改。只要当前数据里能拿到接口状态、IP、默认路由和 DNS，就足够做出总览卡和主连接卡。主连接和总状态可以先由前端推导，后面如果需要再下沉到后端。

首版也不需要做太聪明的规则。把判断做稳，比一开始做一堆半对不对的推理更重要。只要能稳定识别主连接，并把虚拟网卡压到后面，观感就会明显升级。

## 第一步验收感受

做完这一步后，用户第一次打开面板，不该再先看到一长串接口名。

第一眼应该先知道网络现在通不通。

第二眼应该知道当前主要靠哪张网卡联网。

第三眼才是系统里还有哪些其他接口。

如果这三件事都能在第一屏完成，产品味道就已经和原来不是一个层级了。

## 结论

这版真正该补的不是更多功能，而是更强的结论感、角色感和状态感。

只要把首页从接口列表改成网络总览，把主连接从普通列表项提升成核心卡片，再加一点轻诊断，整个侧边栏就会从能用变成有记忆点。
