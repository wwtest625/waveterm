# 卡片终端（Warp 风格）— 产品需求文档（PRD）

## 1. 背景与问题

Wave 现有终端以传统 TTY 视图呈现，适合交互式程序与长时间会话，但在日常“执行命令 → 查看结果 → 执行下一条命令”的工作流中：

- 输出在滚屏里不断叠加，回看信息成本高。
- 很难把一次命令的输入/输出当作一个“可复用的单元”管理（复制、折叠、搜索、分享）。
- 对于非交互命令，传统终端的“光标/屏幕状态机”能力并没有被充分利用。

目标是引入类似 Warp 的 **卡片终端（Card Terminal）**：将每次命令执行封装为一张卡片，像聊天对话一样呈现输入/输出，从而提升可读性与可检索性。

## 2. 目标

1. **对话流展示**：用户输入（右侧气泡）+ 命令输出（左侧气泡），一眼对应。
2. **流式输出**：命令执行中输出实时追加（像聊天正在输出）。
3. **可管理输出**：长输出自动折叠；支持搜索、复制。
4. **状态可见**：显示退出码、耗时；失败显著。
5. **会话语义延续**：cwd（当前目录）在卡片间延续并显示。
6. **上下文/容器标识**：显示并可编辑“上下文标签”（如容器名 / Pod 名），并跨重启/刷新保持。
7. **交互式自动降级**：遇到 TUI/交互式命令时自动切回传统终端视图，保证可用性。

## 3. 非目标

- 不尝试让卡片视图完整模拟 VT 终端（光标定位、全屏刷新、复杂控制序列）。
- 不保证对所有进度条/动态重绘输出在卡片里“完美呈现”。
- 不把卡片历史做成可分享/导出为独立文件的正式功能（可用复制替代）。
- 不支持在卡片模式下可靠运行 vim/tmux/top 等全屏交互式程序（会自动切换到普通终端）。

## 4. 适用范围与入口

### 4.1 适用对象

- 仅针对 **普通 shell 终端块**（`view=term` 且 `controller=shell`）。
- `controller=cmd`（一次性命令块）保持原逻辑，不强制卡片化。

### 4.2 入口与默认行为

- 默认进入卡片模式（`term:mode = cards`）。
- 终端 Header 提供一键切换：**卡片终端 ↔ 普通终端**。
- 进入 Wave App（vdom）后返回时，回到普通终端（避免误回到卡片）。

## 5. 核心交互与信息架构

### 5.1 页面结构

卡片模式由三部分组成：

1. 顶部上下文条（Topbar）
   - 连接名（local / SSH 连接名等）
   - 当前目录（cwd）
   - 上下文标签（可编辑）
   - 搜索框
   - “普通终端”按钮（显式切换）
2. 卡片列表（Timeline）
3. 底部输入条（Inputbar）
   - 多行输入
   - 发送按钮（快捷键 Ctrl+Enter）

### 5.2 卡片结构

每条命令对应 1 张卡片：

- 右侧：命令输入气泡
  - 标题：`$`
  - 元信息：exit code、耗时（完成后显示）
  - 内容：命令文本
- 左侧：输出气泡
  - 标题：输出
  - 操作：复制、展开/收起
  - 内容：按行渲染（保留 ANSI 颜色/样式）

### 5.3 长输出折叠

- 默认最多展示前 N 行（建议 60 行）
- 超过 N 行显示“展开(总行数)”按钮
- 展开后可收起

### 5.4 搜索

- 以关键字过滤卡片：匹配命令文本或输出文本（忽略 ANSI）

### 5.5 复制

- “复制”默认复制 **纯文本输出**（剥离 ANSI 序列）

## 6. 命令生命周期与状态

### 6.1 理想路径（启用 Shell Integration）

依赖 OSC 16162：

- `C`（Command Execution）：命令开始（可带 cmd64）
- `D`（Done）：返回 exit code
- `A`（Prompt Start）：新 prompt 开始，用于判定命令结束

行为：

1. 用户发送命令：立即创建 Pending 卡片（右侧显示输入）
2. 收到 `C`：卡片进入 streaming 状态，开始捕获输出
3. 输出流式追加
4. 收到 `D/A` 后：卡片结束，记录 exit code 与耗时

### 6.2 兜底路径（未启用 Shell Integration）

在未启用 shell integration 时，仍需要“能看到输出”，但命令边界不可靠：

- 发送命令后立刻开始捕获输出
- 若输出停止超过一小段时间（如 650ms）则结束该卡片
- 若较长时间无输出（如 2.5s）则结束该卡片

Topbar 显示提示：未启用 shell integration，卡片分段/退出码可能不完整。

## 7. 交互式命令自动降级

当检测到以下情况，自动切回普通终端：

- 进入 alternate buffer
- 识别到 blocking 命令（vim/tmux/top/less 等）

并在当前 Pending 卡片上追加说明：已进入交互模式，已切换到普通终端。

## 8. 上下文标签（容器/Pod 等）

### 8.1 手动编辑

- 用户可在 Topbar 的“上下文/容器名”输入框编辑。

### 8.2 自动推断（基础版）

- 从命令文本推断：
  - `docker exec <container> ...`
  - `podman exec <container> ...`
  - `kubectl exec <pod> ...`

推断结果会更新上下文标签。

### 8.3 持久化

- 上下文标签写入运行时信息：`RTInfo(term:contextlabel)`

## 9. 关键边界与限制

- ANSI 支持主要覆盖 SGR（颜色/加粗/下划线等），不保证所有控制序列。
- 进度条/回车覆盖（`\r`）可能在卡片里表现为多行（可接受）。
- 单条命令输出过大时需要做上限（例如最多保留最近 2000 行）。

## 10. 验收标准（MVP）

1. 默认卡片模式可用；可一键切回普通终端；Header 可切换。
2. `echo 123`、`ls/dir` 等非交互命令：输出能在卡片里实时显示。
3. 输出支持 ANSI 基本颜色。
4. 长输出可折叠/展开；可复制纯文本。
5. cwd 在 Topbar 显示并随 `cd` 更新。
6. `vim/top/less` 等交互命令：自动切回普通终端。
7. 上下文标签可编辑并持久化（刷新后仍存在）。

## 11. 后续迭代方向

- 输出渲染增强：更完整 ANSI/控制序列支持；更好处理进度条。
- 卡片“命令块”管理：重跑、收藏、分享/导出。
- 全局搜索与跳转；卡片分组（按目录/上下文）。
- 容器/集群上下文自动识别更完善（namespace、context、kubeconfig）。
