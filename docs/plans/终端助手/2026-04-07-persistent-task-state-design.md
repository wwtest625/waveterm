# 持续任务状态设计稿

**当前结论**
这份文档最初想描述的是“后端持久任务链”。但按现在仓库里的实现来看，Wave 还没有那层完整骨架；目前落地的是一组更薄的状态基础：
`lasttaskstate` 这种会话级字符串状态、前端本地的 `AgentRuntimeSnapshot`、命令交互状态、以及工具调用的 `data-tooluse` / `data-toolprogress` 事件。

换句话说，现状已经能“看见并跟踪一次任务运行”，但还不能“把一整条可恢复的任务链持久化下来”。

**已经落地的部分**
`pkg/aiusechat/agentmode.go` 里已经有终端助手的模式定义和一组 P0 验收项，测试明确区分了哪些只是文档化、哪些已经验证；但 `validateToolForAgentMode` 现在仍是空实现，所以工具级强制还没有真正闭环。  
`pkg/aiusechat/usechat.go` 会把工具调用过程写成 `data-tooluse` 事件，并在完成后回写状态；文件编辑工具也已经有小步修改和备份差异的工作流提示。  
前端的 `frontend/app/aipanel/aitypes.ts` 和 `frontend/app/aipanel/waveai-model.tsx` 里已经有本地运行态快照：`idle`、`submitting`、`planning`、`executing`、`awaiting_approval`、`interacting`、`retrying`、`completed`、`failed_*` 等状态会在面板内切换。  
会话元数据也已经有 `LastTaskState` 字段，当前代码里会写入类似 `submitting`、`idle`、`executing` 这样的字符串值，但它还只是一个摘要字段，不是完整步骤图。

**还没落地的部分**
现在还没有一个真正的、后端持久化的 task graph。  
也就是说，以下内容仍然是设计目标，而不是现成实现：

- 多步骤任务的结构化步骤列表
- 每一步的完成/失败/重试关系
- 当前焦点 step 的自动推进
- 刷新或重连后的完整续接
- 模式级工具门禁的真正强制
- 把 `wave_run_command`、`wave_get_command_result`、`write_text_file`、`edit_text_file` 统一挂到同一条持久任务链上

**设计边界**
如果后面要补这一层，建议继续保持轻量，不要先做成通用 todo 系统。  
更合理的方向是把“任务链”限制为少量结构化字段，然后让前端只读这些状态，不自己推导任务语义。

第一版比较适合的最小状态，仍然是：

- 当前焦点步骤
- 步骤列表及状态
- 下一步指针
- 失败时的收窄提示

**当前实现和目标的差距**
当前实现已经具备“单次运行的运行态展示”和“工具调用后状态回写”的底座。  
缺口在于：这些状态还分散在会话摘要、前端内存状态、工具消息流和命令交互状态里，彼此之间没有统一的持久任务模型。

**后续方向**
如果继续推进这条线，下一步应先定义统一的 task state 数据结构，再把命令执行、结果读取、文件编辑和交互状态都映射到这一个模型里。  
前端任务卡可以继续保持薄，只负责渲染和恢复，不负责调度和推理。

**验收口径**
等这层真正做完后，文档里的验收标准应该变成：

- 面板能稳定显示当前运行态
- 任务状态能在刷新后恢复
- 工具成功/失败能驱动同一条状态链
- 失败时能给出更窄的重试提示
- 前端不再依赖散落的消息块去猜任务进度
